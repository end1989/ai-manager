# 📦 One-Shot Prompt — Build the AI Manager Infrastructure (Python)

**ROLE (system/developer):**
You are a **principal engineer** tasked with shipping a **local-first AI Manager infrastructure** in Python. The product is a working codebase the user can clone, run, and extend. Your focus is the **Manager** (control plane): task planning, delegation, worker lifecycle, evidence-based review, and safe execution. You must generate a complete repository with code, tests, docs, CI, and runnable demos.

---

## 0) Goals (what to build)

* A **Manager service** (FastAPI + orchestration core) that:

  * Accepts a **Task** (spec) via REST/CLI.
  * **Plans** tasks (can be manual today, pluggable for LLM planning later).
  * **Delegates** work to **Workers** via a local runner (subprocess) with an isolated venv.
  * **Reviews** worker outputs skeptically using rules + tests + static checks.
  * Stores everything (tasks, runs, artifacts, logs) in **SQLite**.
* A **Worker runtime** that a Manager can spawn:

  * Executes a “Worker Prompt” plan locally (no external APIs required).
  * Produces **Task Reports** + **PR Proposals** (JSON) + test evidence.
* A **Task Queue** (SQLite + simple FIFO) and orchestration loop.
* **Skeptical review engine**: run tests, lint, mypy, and basic policy checks before “merge”.
* **Local-only** by default (no secrets needed). Provide extension points for real LLMs later.
* Ship with:

  * **CLI** (`manager`, `worker`) and **FastAPI** admin UI (/docs).
  * **Unit + integration tests** (pytest) and example scenario.
  * **Dockerfile**, **docker-compose**, **GitHub Actions CI**.
  * **Docs**: README, ARCHITECTURE.md, DECISIONS.md, CONTRIBUTING.md.

## 1) Stack & Standards

* **Python 3.11+**.
* Packaging: `pyproject.toml` (uv/poetry-compatible).
* Web: **FastAPI** + **Uvicorn**.
* DB: **SQLite** via **SQLModel** (or SQLAlchemy + Pydantic).
* Tests: **pytest**, **pytest-cov**; coverage ≥ **85% on changed lines**.
* Lint/Format/Types: **ruff**, **black**, **mypy**.
* Logging: structured (JSON in prod), with request IDs and run IDs.
* Config: `pydantic-settings`; 12-factor style.
* Container: minimal **Dockerfile**, **docker-compose.yml** for dev.
* CI: GitHub Actions running lint, type checks, tests.

## 2) Architecture (deliver these modules)

* `core/manager.py` — orchestrator: plan → enqueue → dispatch → review → merge.
* `core/planner.py` — slices big tasks to subtasks (rule-based; LLM plugin hook).
* `core/queue.py` — SQLite-backed queue, status transitions, idempotency keys.
* `core/review.py` — skeptical reviewer: executes test suite, static checks, policy rules; emits **Review Report**.
* `core/artifacts.py` — run directories, logs, JSON artifacts, retention policy.
* `core/executor.py` — worker subprocess launcher; ephemeral **venv** per run; timeouts; resource caps.
* `core/schemas.py` — **Task Spec**, **Task Report**, **PR Proposal**, **Review Report** (Pydantic).
* `adapters/worker_prompt.py` — baseline Worker behavior (no real LLM required).
* `adapters/llm_stub.py` — stub that returns deterministic text (future: swap with real LLM).
* `api/http.py` — FastAPI routes (submit task, list tasks, runs, artifacts; trigger review).
* `cli/manager_cli.py` — `manager` commands (submit, plan, run, review, logs).
* `cli/worker_cli.py` — `worker` command to execute an assigned task spec locally.
* `store/models.py` — SQLModel entities (Task, Run, Artifact, Review, Event).
* `ops/` — Dockerfile, compose, CI, Makefile, logging config, seed demo.

**Security & Safety:**

* Workers run with **no network** by default (configurable), **low privileges**, **timeouts**, and **cwd sandbox** (`runs/<run-id>`).
* Max artifact sizes; redact env; prevent secret leakage in logs.

## 5) Features & Behaviors to Implement

### Manager Control Plane

* **Submit Task** (CLI + POST `/tasks`): persist task, enqueue job.
* **Dispatcher Loop**: pull next pending task → create run → spawn worker subprocess:

  * Create isolated `runs/<run-id>`; write **Task Spec** JSON.
  * Create ephemeral venv; install project (editable) + test deps.
  * Launch `worker` entrypoint with `--task file.json` and resource limits.
  * Capture stdout/stderr → `artifacts/`.
* **Collect Results**: parse **Worker Task Report** + **PR Proposal**; persist.
* **Skeptical Review**:

  * Run `ruff`, `black --check`, `mypy`, `pytest --cov`.
  * Check coverage delta for changed files.
  * Validate schemas; verify no network used (if `NO_NET=1`).
  * Emit **Review Report**.
* **Merge Simulation**:

  * If approved, write “merged” diff into working tree under `services/…`.
  * Update CHANGELOG; log event.

### Worker Runtime

* Read **Task Spec**; implement minimum code in a sandbox subfolder (e.g., `workdir/`).
* Generate tests first for bugfix tasks or minimal feature tests for new components.
* Produce **Worker Task Report** + **PR Proposal**; write to `artifacts/`.
* Never claim success without evidence (pytest summary, coverage, linter output).

### Task Queue & Persistence

* Tables: `tasks`, `runs`, `artifacts`, `prs`, `reviews`, `events`.
* Status graph: `queued → running → awaiting_review → approved|changes_requested|rejected → merged`.

### Config & Safety

* Config via env with defaults; example: `MANAGER_DB_URL`, `RUN_CPU_SECS`, `RUN_MEM_MB`, `NO_NET`.
* Timeouts on worker; kill/cleanup on overrun; exit codes mapped to run status.
* Log every state transition with `run_id`, `task_id`, `pr_id`.
